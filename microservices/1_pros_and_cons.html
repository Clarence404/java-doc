优势与挑战

2025/2/15大约 3 分钟

优势与挑战

一、单体应用的瓶颈

随着业务增长，单体应用逐渐暴露以下问题：

小团队早期              业务扩张后
┌────────────────┐      ┌────────────────────────────────┐
│  简单单体应用    │  →   │  数百万行代码的"大泥球"           │
│  易开发易部署    │      │  启动耗时 10+ 分钟               │
│  IDE 友好       │      │  任何改动都要全量部署             │
└────────────────┘      │  一个模块 OOM 拖垮整个进程        │
                        │  新人需要几个月才能理解全貌        │
                        └────────────────────────────────┘
问题具体表现
复杂度失控代码量膨胀，任何人都无法完整理解整个系统
部署风险高改一行代码需要全量打包部署，风险难以隔离
扩展受限无法对单个热点模块单独扩容，只能整体扩展
技术债积累难以引入新技术，框架升级影响全局
团队协作瓶颈多团队共享一个代码仓库，合并冲突频繁

二、微服务的优势

优势说明
独立部署每个服务可以独立发布，缩短发布周期，降低单次发布风险
独立扩展秒杀服务扩 10 倍，不影响其他服务；可按需选择硬件规格
技术异构订单服务用 Java，推荐服务用 Python，各取所长
故障隔离通知服务宕机不影响核心下单流程（需配合熔断降级）
团队自治每个团队对自己的服务完全负责，减少协调成本
易于理解单个服务代码量小，新人上手快

三、微服务的挑战

挑战说明应对方案
分布式复杂性网络调用可能超时、失败，比本地调用复杂得多熔断、重试、超时设计
数据一致性跨服务事务无法用本地事务保证Saga 模式、最终一致性
服务间调用链一个请求经过多个服务，排查问题困难链路追踪（SkyWalking）
运维复杂度几十个服务需要监控、部署、扩缩容容器化（Docker + K8s）
测试困难集成测试需要启动多个服务或 stubConsumer-Driven Contract
接口版本管理服务间接口变更需要兼容旧版本语义化版本 + 向后兼容原则

结论：微服务不是银弹。没有完善的 CI/CD、容器化基础设施和监控体系，微服务带来的麻烦远大于收益。

四、何时该从单体迁移到微服务

评估维度                    建议
─────────────────────────────────────────
团队 < 8 人              → 先别拆，单体够用
业务边界模糊              → 先梳理领域模型再拆
没有 Docker/K8s          → 先补基础设施
没有完善监控              → 先补可观测性
单体已影响发布效率         → 可以开始规划拆分

五、单体到微服务迁移策略

策略一：停止挖掘

停止让单体继续变大。新功能直接开发为独立微服务，不再往旧单体里加代码。

策略二：前后端分离

将表现层与业务逻辑、数据访问层分开，形成可独立部署的两个应用——这是减小单体复杂度的第一步。

策略三：绞杀者模式（Strangler Fig）

逐步抽取模块为独立微服务，不做大爆炸式重写：

① 在单体前加 API Gateway，所有流量经网关
② 将某一模块抽取为微服务，网关将对应流量路由到新服务
③ 重复②，逐步"绞杀"单体
④ 单体功能全部迁移完成，下线单体

优先抽取顺序：频繁变化的模块 → 资源消耗大的模块 → 边界最清晰的模块

"the only thing a Big Bang rewrite guarantees is a Big Bang!" — Martin Fowler
