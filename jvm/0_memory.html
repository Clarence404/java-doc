JVM 内存结构

一、运行时数据区

┌─────────────────────────────────────────────────┐
│                  JVM 运行时数据区                  │
│                                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │ 程序计数器 │  │ 虚拟机栈  │  │ 本地方法栈 │  ← 线程私有 │
│  └──────────┘  └──────────┘  └──────────┘      │
│  ┌─────────────────────────────────────────┐    │
│  │              堆（Heap）                  │    │
│  │   年轻代（Eden + S0 + S1）  │  老年代     │  ← 线程共享 │
│  └─────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────┐    │
│  │           方法区（MetaSpace）             │  ← 线程共享 │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
  直接内存（堆外，不在 JVM 规范内）

程序计数器

  • 每个线程私有，记录当前线程执行的字节码行号
  • 唯一不会发生 OOM 的区域
  • 执行 Native 方法时值为 Undefined

Java 虚拟机栈

  • 线程私有，管理 Java 方法调用，每个方法对应一个栈帧
  • 栈帧结构：局部变量表 / 操作数栈 / 动态链接 / 方法出口 → 详见 2_bytecode
  • 异常：StackOverflowError（栈深度超限）、OutOfMemoryError（栈扩展失败）
  • 参数：-Xss512k（每线程栈大小）

本地方法栈

  • 为 Native 方法服务，HotSpot 将其与虚拟机栈合二为一

堆

  • 所有线程共享，存放对象实例
  • 分代：年轻代（Eden + Survivor×2）+ 老年代
  • 异常：OutOfMemoryError: Java heap space
  • 参数：-Xms（初始）/ -Xmx（最大），建议两者相等避免动态扩容

方法区（MetaSpace）

  • 存储类结构信息、常量、静态变量、即时编译代码
  • JDK 8 用元空间（本地内存）替代永久代，默认无上限
  • 异常：OutOfMemoryError: Metaspace
  • 参数：-XX:MetaspaceSize=256m / -XX:MaxMetaspaceSize=512m

直接内存

  • 堆外内存，通过 ByteBuffer.allocateDirect() 或 Unsafe.allocateMemory() 分配
  • 不受 JVM 堆限制，NIO 操作减少内核态/用户态数据复制
  • 参数：-XX:MaxDirectMemorySize=1g

二、对象内存布局

HotSpot 中一个对象在堆中的存储结构：

┌────────────────────────────────┐
│          对象头（Header）        │
│  ┌─────────────────────────┐   │
│  │  Mark Word（8字节）      │   │  哈希码、GC年龄、锁状态
│  ├─────────────────────────┤   │
│  │  Klass Pointer（4/8字节）│   │  指向类元数据（方法区）
│  └─────────────────────────┘   │
├────────────────────────────────┤
│       实例数据（Instance Data）  │  字段值，父类字段在前
├────────────────────────────────┤
│       对齐填充（Padding）        │  补齐至 8 字节整数倍
└────────────────────────────────┘

Mark Word 在不同锁状态下的内容（32位简化）：

锁状态Mark Word 内容
无锁hashCode（25位）+ 分代年龄（4位）+ 偏向标志(0) + 01
偏向锁线程ID（23位）+ Epoch（2位）+ 分代年龄（4位）+ 偏向标志(1) + 01
轻量级锁指向栈中锁记录的指针 + 00
重量级锁指向 Monitor 对象的指针 + 10
GC标记空 + 11

压缩指针（-XX:+UseCompressedOops，默认开启）：64位 JVM 中 Klass Pointer 压缩为 4 字节，堆 < 32GB 时有效。

三、TLAB（Thread Local Allocation Buffer）

  • 线程私有的分配缓冲区，减少多线程在 Eden 区分配时的 CAS 竞争
  • 默认大小为 Eden 区的 1%，参数：-XX:TLABWasteTargetPercent
  • 分配流程：TLAB 分配 → TLAB 不足则 Eden 区 CAS 分配 → Eden 满则触发 Minor GC

四、内存分配策略

策略说明
对象优先在 Eden 区分配大多数新对象在年轻代 Eden
大对象直接进入老年代超过 -XX:PretenureSizeThreshold 的对象
长期存活对象晋升老年代经历 -XX:MaxTenuringThreshold（默认15）次 Minor GC
动态年龄判断Survivor 区同年龄对象总和 > Survivor 一半，该年龄对象直接晋升
空间分配担保Minor GC 前检查老年代最大可用连续空间 > 新生代所有对象总和

五、String 常量池

位置变迁

JDK 版本String 常量池位置说明
JDK 6 及之前永久代（PermGen）大小固定，intern() 滥用易 OOM
JDK 7+堆（Heap）随堆一起 GC，intern() 返回堆中已有对象的引用
JDK 8+堆（Heap）同上，PermGen 改为 Metaspace

String.intern()

intern() 方法将字符串放入常量池并返回常量池中的引用：

String s1 = new String("hello");   // 堆中新对象
String s2 = s1.intern();           // 将 "hello" 放入常量池，返回常量池引用
String s3 = "hello";               // 直接引用常量池

System.out.println(s1 == s2);  // false（s1 是堆对象，s2 是常量池引用）
System.out.println(s2 == s3);  // true（都指向常量池同一对象）

使用场景：大量重复字符串（如日志字段、枚举值），手动 intern() 可节省内存；但过度使用会增加常量池查找开销。

G1 字符串去重（String Deduplication）

# JDK 8u20+ G1 专属优化，自动合并堆中内容相同的 String char[] 数组
-XX:+UseG1GC -XX:+UseStringDeduplication

与 intern() 的区别：去重只合并底层 char[]（或 JDK 9+ 的 byte[]），不影响 String 对象本身；intern() 合并的是 String 对象引用。

六、常见面试问题

Q：堆和方法区的区别？ 堆存放对象实例（运行时数据），方法区存放类的结构信息（元数据）。两者都是线程共享区域。

Q：Java 对象一定在堆上分配吗？ 不一定。JIT 的逃逸分析可能将未逃逸对象优化为栈上分配或标量替换，这样对象不在堆上。详见 6_jit。

Q：为什么 JDK8 要用元空间替代永久代？ 永久代大小固定且受 JVM 内存限制，容易 OOM；元空间使用本地内存，可动态扩容，且便于 HotSpot 与 JRockit 的合并。
