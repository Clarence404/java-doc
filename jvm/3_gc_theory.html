GC 基础理论

一、如何判断对象可以被回收

引用计数法（已淘汰）

每个对象维护一个引用计数器，计数为 0 时回收。 缺陷：无法解决循环引用（A 引用 B，B 引用 A，两者计数都不为 0 却都不可达）。

可达性分析（JVM 采用）

从 GC Roots 出发，沿引用链遍历，不可达的对象即为垃圾。

GC Roots 包括：

  • 虚拟机栈中引用的对象（局部变量）
  • 方法区中静态字段引用的对象
  • 方法区中常量引用的对象（字符串常量池）
  • 本地方法栈中 JNI 引用的对象
  • JVM 内部引用（基本类型的 Class 对象、常驻异常对象）
  • 被 synchronized 持有的对象

二、引用类型

引用类型类GC 行为典型用途
强引用普通 = 赋值永不回收所有正常对象
软引用SoftReference内存不足时回收缓存（Caffeine 用到）
弱引用WeakReference下次 GC 时回收WeakHashMap、ThreadLocal Entry
虚引用PhantomReference随时回收，get() 返回 null跟踪对象被回收的时机

三、GC 算法

标记-清除（Mark-Sweep）

标记阶段：从 GC Roots 出发，标记所有可达对象
清除阶段：回收未被标记的对象
  • 优点：实现简单
  • 缺点：内存碎片，大对象分配困难；标记和清除都需要暂停应用

复制（Copying）

将内存分为两块，每次只用一块
GC 时将存活对象复制到另一块，清空当前块
  • 优点：无碎片，分配速度快（指针碰撞）
  • 缺点：内存利用率 50%
  • 应用：年轻代（Eden + S0 + S1，8:1:1，实际利用率 90%）

标记-整理（Mark-Compact）

标记阶段：同标记-清除
整理阶段：将存活对象向一端移动，清除边界外内存
  • 优点：无碎片
  • 缺点：移动对象开销大，需要更新所有引用
  • 应用：老年代

分代收集（Generational）

基于"弱分代假说"（大多数对象朝生夕死）和"强分代假说"（熬过越多 GC 的对象越难消亡）：

年轻代（Young Generation）
├── Eden 区：新对象诞生处（约 80%）
├── Survivor 0（S0，约 10%）
└── Survivor 1（S1，约 10%）
    → Minor GC：复制算法，频繁，STW 短

老年代（Old Generation）
    → Major/Full GC：标记整理/清除，不频繁，STW 长

对象晋升流程：

  1. 新对象先进 Eden
  2. Minor GC 后存活对象进 Survivor，年龄 +1
  3. 年龄达到阈值（默认 15）或 Survivor 空间不足 → 晋升老年代

四、GC 触发时机

Minor GC（Young GC）

  • Eden 区满，无法为新对象分配空间
  • TLAB 分配失败

Full GC

  • 老年代空间不足（对象晋升 or 大对象直接入老年代）
  • 方法区（MetaSpace）空间不足
  • 显式调用 System.gc()（建议 JVM 执行，不保证）
  • Minor GC 前空间分配担保失败
  • CMS 并发模式失败（退化为 Serial Old）

五、STW（Stop The World）

GC 期间所有应用线程暂停，只有 GC 线程运行。

STW 是必要的：可达性分析需要对象图快照（snapshot），若分析过程中对象引用关系还在变化，结果不可靠（误回收存活对象 → 悬空指针）。

缩短 STW 的技术：

技术说明
并发标记GC 线程与应用线程并发执行标记，只有初始/最终标记需 STW
增量更新（CMS）记录并发标记期间的引用变化，重新标记时修正
原始快照（SATB）G1/ZGC 使用，并发标记开始时保存对象图快照
染色指针（ZGC）将 GC 信息编码在指针中，无需扫描整堆即可知道对象状态
读屏障（ZGC）对象访问时拦截，实现并发重定位，STW < 1ms

六、常见面试问题

Q：GC Roots 有哪些？为什么这些对象是根？ 这些对象本身不需要被扫描就能确定"存活"（线程在用的变量、静态字段等），从它们出发可以找到所有被程序真正使用的对象。

Q：Minor GC 会扫描老年代吗？ 会扫描一部分。老年代对象可能引用年轻代对象（跨代引用），JVM 用记忆集（Remembered Set）/ 卡表（Card Table）记录跨代引用，Minor GC 只需扫描脏卡，避免全堆扫描。

Q：System.gc() 一定会触发 Full GC 吗？ 不一定。System.gc() 只是"建议"JVM 执行 GC，JVM 可以忽略。-XX:+DisableExplicitGC 可完全禁用显式 GC 调用（Netty 等框架中常用，防止 DirectBuffer 被误回收导致性能下降）。
