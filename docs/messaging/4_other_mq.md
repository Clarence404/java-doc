# MQ 选型指南

---

## 一、场景决策表

| 场景 | 推荐 | 原因 |
|------|------|------|
| 日志收集 / 埋点上报 | **Kafka** | 高吞吐、顺序写、消息可回放，Flink/Spark 直接消费 |
| 大数据 Pipeline / 事件溯源 | **Kafka** | 消息长期保留，支持多消费者组独立消费同一份数据 |
| 电商核心链路（订单/支付/库存）| **RocketMQ** | 事务消息原生支持，延迟消息（超时取消），国内维护活跃 |
| 分布式事务最终一致 | **RocketMQ** | 半消息 + 本地事务 + 回查机制，开箱即用 |
| 中低并发业务解耦 | **RabbitMQ** | Spring AMQP 集成简单，灵活路由（Topic/Fanout），运维成本低 |
| 需要对接外部系统 | **RabbitMQ** | 标准 AMQP/MQTT 协议，兼容性最好 |
| 物联网设备消息 | **RabbitMQ** (MQTT) / **Kafka** | RabbitMQ 支持 MQTT；大规模设备用 Kafka |
| 超大规模流式处理 | **Kafka** | Kafka Streams / Kafka Connect 生态最完善 |
| 云原生 / 存算分离 | **Pulsar** | 计算层（Broker）与存储层（BookKeeper）独立扩展 |

## 二、关键维度速查

| 维度 | Kafka | RocketMQ | RabbitMQ |
|------|-------|----------|---------|
| 吞吐量 | 极高 | 高 | 中等 |
| 延迟 | ms 级 | ms 级 | µs 级 |
| 事务消息 | 有限 | ✅ 原生 | 有限 |
| 延迟消息 | ❌ 不原生 | ✅ 18 级 | TTL+DLX |
| 消息回放 | ✅ | ✅ | ❌ |
| 死信队列 | 需自行实现 | ✅ | ✅ |
| 运维难度 | 高 | 中 | 低 |
| 适合团队 | 大数据/平台 | 国内业务后端 | 传统企业/中小团队 |

## 三、Apache Pulsar（了解）

Pulsar 是 Apache 开源的新一代消息流平台，兼具消息队列和流处理能力，核心特点是**存算分离**：

```
Producer → Broker（无状态，纯计算层）→ BookKeeper（持久化存储层）
              ↑
           Broker 可随意扩缩容，不影响已存储的数据
```

| 特性 | 说明 |
|------|------|
| **存算分离** | Broker 无状态，存储由 BookKeeper 独立管理，各自独立扩缩容 |
| **多租户** | 原生 Namespace + Topic 隔离，单集群支持多团队 |
| **消息模型** | 同时支持队列（Consumer Group）和流（Subscription）|
| **Geo 复制** | 原生跨机房/跨地域复制 |
| **适用场景** | 云原生平台、需要跨地域同步、超大规模多租户场景 |

**当前现状**：Pulsar 在国内大厂（腾讯、滴滴）有生产应用，但生态成熟度和社区活跃度仍低于 Kafka 和 RocketMQ，新项目不建议贸然引入。

## 四、常见误区

| 误区 | 正解 |
|------|------|
| "Kafka 就是最好的 MQ" | Kafka 不支持延迟消息、事务消息，也不适合小消息量高可靠业务 |
| "RabbitMQ 性能差" | 微秒级延迟，适合中低并发场景，不适合大数据量是吞吐限制，不是缺陷 |
| "换 MQ 很简单" | 消息语义（顺序/事务/回放）差异大，换 MQ 通常需要改消费逻辑 |
| "MQ 解决了分布式事务" | MQ 只实现最终一致，中间状态仍需业务侧幂等和补偿 |
