# 消息中间件总览

消息中间件（Message Queue，MQ）是分布式系统中的异步通信层，解耦服务依赖，缓冲流量洪峰，提升系统弹性。

---

## 一、核心作用

### 1、应用解耦

服务之间通过 MQ 通信，下游故障不影响上游，新增下游系统也无需修改上游代码。

```
订单系统 ──→ MQ ──→ 库存系统
                └──→ 积分系统
                └──→ 物流系统
```

### 2、流量削峰

高峰期请求先写入 MQ，消费者按自身处理能力匀速消费，防止突发流量打垮后端。

```
5000 req/s → MQ → 消费者（处理能力 2000/s）→ 积压慢慢消化
```

### 3、异步处理

主流程只做核心操作（写 DB），耗时的旁路操作（发邮件、推送、更新统计）异步由消费者完成，降低响应延迟。

---

## 二、消息模型

| 模型 | 说明 | 典型应用 |
|------|------|---------|
| **点对点（P2P）** | 消息只被一个消费者处理，消费后删除 | 任务队列、订单处理 |
| **发布/订阅（Pub/Sub）** | 消息被所有订阅者接收，可重放 | 事件通知、日志收集 |
| **顺序消息** | 同业务 key 的消息严格按序消费 | 订单状态流转、账务流水 |
| **延迟消息** | 消息在指定时间后才投递 | 订单超时取消、定时提醒 |
| **事务消息** | 本地事务与消息投递原子性一致 | 分布式事务最终一致 |

---

## 三、通用可靠性保障

消息不丢失需要三层协同保障：

| 层次 | 措施 | 说明 |
|------|------|------|
| **生产者** | 发送确认（ACK / Confirm）| 收到 Broker 确认后才认为发送成功；失败则重试 |
| **Broker** | 消息持久化 + 主从同步 | 写入磁盘（同步/异步刷盘）；多副本复制防止单点丢失 |
| **消费者** | 手动 ACK | 业务处理成功后才通知 Broker 删除/偏移，处理失败则重投 |

### 幂等消费（防重复处理）

网络重试和消费者 Rebalance 都可能导致同一条消息被消费多次，业务层需保证幂等：

```java
// 方案一：Redis SETNX（推荐，高性能）
boolean isNew = redis.opsForValue()
    .setIfAbsent("msg:consumed:" + msgId, "1", Duration.ofHours(24));
if (!isNew) return;  // 已处理，跳过

// 方案二：DB 唯一索引（适合需要事务保证的场景）
// INSERT IGNORE INTO msg_consumed(msg_id) VALUES(?)
// 抛出唯一冲突异常 → 幂等返回
```

### 死信队列（DLQ）

消息处理失败超过重试次数后，进入死信队列，防止一直阻塞正常消费：

```
消费失败 → 重试 N 次 → 超过上限 → 死信队列（DLQ）
                                      ↓
                             人工排查 / 告警 / 补偿处理
```

## 四、三大 MQ 对比

| 维度 | Kafka | RocketMQ | RabbitMQ |
|------|-------|----------|---------|
| **定位** | 分布式事件流平台 | 分布式消息中间件 | 传统消息代理 |
| **协议** | 自定义 | 自定义 | AMQP / MQTT / STOMP |
| **吞吐量** | 极高（百万/s）| 高（十万/s）| 中等（万/s）|
| **延迟** | 较低（毫秒级）| 低（毫秒级）| 极低（微秒级）|
| **顺序消息** | 分区内有序 | 支持局部顺序 | 队列内有序 |
| **事务消息** | 有限（Exactly Once）| 原生支持 | 支持（较复杂）|
| **延迟消息** | 不原生支持 | 原生支持（18 级）| TTL + DLX 实现 |
| **消息存储** | 长期保留（日志文件）| 可配置保留 | 消费后删除（默认）|
| **消息回放** | ✅ 支持（按 Offset）| ✅ 支持 | ❌ 不支持 |
| **高可用** | ISR + Leader 选举 | 主从 + DLedger | 镜像队列 / Quorum |
| **运维复杂度** | 高（需 ZK 或 KRaft）| 中（需 NameServer）| 低（单机简单）|
| **生态** | Kafka Connect/Streams，大数据完善 | 阿里云完善，国内金融电商广泛 | Spring AMQP 成熟 |
| **适合场景** | 日志/埋点/大数据管道/事件溯源 | 业务消息/分布式事务/电商核心 | 传统业务解耦/中低并发 |

---

## 五、选型建议

| 场景 | 推荐 | 原因 |
|------|------|------|
| 日志收集 / 埋点 / 大数据 Pipeline | **Kafka** | 高吞吐、消息可回放、Flink/Spark 生态 |
| 电商核心业务（订单/支付/库存）| **RocketMQ** | 事务消息原生支持、延迟消息、国内维护活跃 |
| 中低并发业务解耦 / 微服务通知 | **RabbitMQ** | Spring AMQP 集成简单、灵活路由、运维成本低 |
| 对接外部系统（需标准协议）| **RabbitMQ** | AMQP/MQTT 兼容性好 |
| 超大规模流式处理 | **Kafka** | Kafka Streams / Connect 生态最完善 |
