参考链接：[advanced-java 高可用架构](https://gitee.com/Doocs/advanced-java#%E9%AB%98%E5%8F%AF%E7%94%A8%E6%9E%B6%E6%9E%84)

## 一、什么是高可用

高可用（High Availability，HA）是指系统在**故障、峰值流量、依赖失败**等异常情况下，仍能持续对外提供服务的能力。核心衡量指标为 **SLA（Service Level Agreement）**。

### SLA 可用性等级

| SLA | 可用率 | 年故障时间 | 月故障时间 |
|-----|--------|-----------|-----------|
| 99% | 99% | ≈ 3.65 天 | ≈ 7.3 小时 |
| 99.9%（3 个 9）| 99.9% | ≈ 8.77 小时 | ≈ 43.8 分钟 |
| 99.99%（4 个 9）| 99.99% | ≈ 52.6 分钟 | ≈ 4.4 分钟 |
| 99.999%（5 个 9）| 99.999% | ≈ 5.26 分钟 | ≈ 26 秒 |

> 互联网核心服务通常要求 **4 个 9**，金融/通信等关键系统要求 **5 个 9**。

## 二、高可用核心策略

```
故障预防                  故障容忍                   故障恢复
  ↓                          ↓                          ↓
限流 / 熔断 / 降级         冗余副本 / 多活部署         自动重试 / 快速回滚
负载均衡 / 超时控制         舱壁隔离 / 异步解耦         健康检查 / 容器自愈
```

### 常见故障场景与应对

| 故障类型 | 典型场景 | 应对策略 |
|---------|---------|---------|
| 流量洪峰 | 大促秒杀、突发热点请求 | 限流 / 异步削峰 / 预热 |
| 依赖故障 | 下游服务超时 / 不可用 | 熔断 / 降级 / Fallback |
| 单点故障 | 服务实例崩溃 | 多副本 / 故障转移 |
| 雪崩效应 | 一个慢服务拖垮整条链路 | 隔离 + 超时 + 熔断三件套 |
| 慢请求堆积 | DB 慢查询 / 连接池耗尽 | 超时控制 / 舱壁隔离 |

## 三、多活架构

| 架构模式 | 说明 | 适用场景 |
|---------|------|---------|
| **同城双活** | 同城两机房各自承载流量，数据实时同步 | 可接受极低延迟数据同步 |
| **异地多活** | 多地机房同时对外服务，就近接入 | 用户分布广，对延迟敏感 |
| **两地三中心** | 同城双活 + 异地容灾中心 | 金融/政务等强监管场景 |

**异地多活核心挑战**：数据一致性。通常通过**用户分区**（按 userId/地区路由）隔离数据写入范围，避免跨地域写冲突。

## 四、混沌工程

通过**主动注入故障**（网络延迟、实例宕机、CPU 压满）发现系统脆弱点，在真实故障来临前提升系统韧性。

| 工具 | 说明 |
|------|------|
| Chaos Monkey | Netflix 开源，随机终止生产服务实例 |
| ChaosBlade | 阿里开源，支持 Java / OS / 容器层故障注入 |
| Chaos Mesh | CNCF 项目，Kubernetes 原生混沌工程平台 |
