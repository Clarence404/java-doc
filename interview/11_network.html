开发总结-网络协议

精华提炼，细节详见 protocols/

一、TCP 三次握手为什么不能是两次？

三次握手流程：

客户端                          服务端
  ──── SYN(seq=x) ────────────→
  ←─── SYN+ACK(seq=y, ack=x+1) ──
  ──── ACK(ack=y+1) ──────────→
              连接建立

为什么不能是两次：两次握手无法确认客户端的接收能力，且无法防止历史连接的干扰。

经典场景：客户端发了一个 SYN，网络延迟很久才到达服务端。此时客户端已经重发了新 SYN 并建立了新连接。旧的 SYN 到达服务端后，两次握手的话服务端会认为这是新连接，白白浪费资源并造成混乱。三次握手时，客户端收到服务端对旧 SYN 的响应，发现 ack 对不上，会发 RST 关闭这个无效连接。

二、TCP 四次挥手为什么是四次而不是三次？

四次挥手流程：

主动关闭方                      被动关闭方
  ──── FIN ───────────────────→
  ←─── ACK ────────────────────
      （被动方还可能有数据要发）
  ←─── FIN ────────────────────
  ──── ACK ───────────────────→
  [主动方等待 2MSL 后彻底关闭]

为什么四次：TCP 是全双工的，关闭连接需要双方各自关闭自己这个方向。被动方收到 FIN 后，可能还有数据没发完，所以 ACK 和 FIN 无法合并（不像握手时 SYN+ACK 可以合并）。

2MSL 等待的原因：确保最后一个 ACK 能到达对端。若 ACK 丢失，对端会重发 FIN，主动方在 2MSL 时间内仍能响应。

三、TIME_WAIT 和 CLOSE_WAIT 过多怎么排查？

查看连接状态：

netstat -an | awk '/^tcp/ {print $6}' | sort | uniq -c | sort -rn

TIME_WAIT 过多：

  • 原因：服务端主动关闭大量短连接（如 HTTP/1.0 无 Keep-Alive）
  • 后果：端口被占用，新连接建立失败
  • 解决：
    • 开启 tcp_tw_reuse=1（允许将 TIME_WAIT 状态的连接用于新连接）
    • 使用长连接，减少连接频繁建立和断开
    • 设置 SO_REUSEADDR

CLOSE_WAIT 过多：

  • 原因：程序收到对端 FIN 后，没有调用 close()，连接一直处于半关闭
  • 后果：文件描述符泄露，最终无法建立新连接
  • 解决：检查代码资源释放（try-with-resources / finally 块），排查异常路径是否遗漏关闭

四、HTTP/2 相比 HTTP/1.1 有哪些改进？

特性HTTP/1.1HTTP/2
传输格式文本二进制帧（更高效解析）
连接复用管道化（有队头阻塞）多路复用（多 Stream 并发，互不阻塞）
头部压缩每次请求全量发送HPACK 压缩，共享头部字典
服务器推送❌✅ 服务端主动推资源
队头阻塞TCP + HTTP 双层消除 HTTP 层阻塞，TCP 层仍存在

HTTP/3 基于 QUIC（UDP），彻底消除 TCP 队头阻塞，并支持 0-RTT 建连。

五、HTTPS 握手过程是怎样的？

TLS 1.3 握手（简化）：

客户端                                服务端
  ── ClientHello（支持的算法、随机数）──→
  ←── ServerHello（选定算法）+证书 ──────
  ←── CertificateVerify + Finished ──
  客户端验证证书链（根 CA → 中间 CA → 服务端证书）
  双方基于 ECDHE 生成会话密钥
  ── Finished ──────────────────────→
  ←── Application Data（加密通信开始）──

证书验证：浏览器内置根 CA 证书，逐级验证签名，确认服务端身份。

中间人攻击防范：证书绑定了域名，中间人无法伪造合法证书（私钥不在外部流通）。

状态码含义常见场景

七、DNS 解析流程是怎样的？

浏览器缓存 → 系统 hosts 文件 → 本地 DNS 服务器（运营商）
  → 根 DNS（返回顶级域服务器地址）
  → 顶级域 DNS（如 .com，返回权威服务器地址）
  → 权威 DNS（返回最终 IP）
  → 本地 DNS 缓存结果，返回给浏览器

TTL：权威 DNS 可以设置 TTL，控制本地 DNS 缓存时间。CDN 切换、机房迁移时需注意 TTL 生效延迟。

八、TCP 和 UDP 的区别？各自适用什么场景？

TCPUDP
连接面向连接（三次握手）无连接
可靠性可靠（重传、排序、校验）不可靠，可能丢包乱序
速度较慢（有确认机制）快
头部开销20 字节8 字节
适用场景HTTP、数据库、文件传输视频直播、游戏、DNS 查询、QUIC

UDP 为什么适合视频直播：丢几帧画面用户可接受，但如果等重传再播放，延迟会让体验更差。应用层自己做简单纠错比 TCP 更合适。
